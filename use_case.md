# Юзкейсы AI-native биллинга: подробное описание

**Статус:** v0.1, детализация к `billing_aggregates.md` **Назначение документа:** развернуть кратко описанные в карте агрегатов юзкейсы (UC-1…UC-10) до уровня конкретных сценариев — с примерами данных, потоком команд по агрегатам, событиями и инвариантами, которые при этом защищаются.

Условные обозначения:

- `→` — переход через доменное событие (eventual, отдельная транзакция)

- `⇒` — команда внутри одной транзакции над одним агрегатом

## UC-1/2. Постановка и валидация тарифа

**Контекст.** Маркетолог приносит текст: *«Для тарифа "Комфорт" с 1 июля 2026 действует базовая ставка 3,20 ₽/кВт·ч, а при потреблении свыше 300 кВт·ч в месяц — надбавка 15% на превышение»*. AI-агент формализует это в DSL.

**Затронутые агрегаты:** только `TariffVersion`.

### Поток

1. `TariffVersion.DraftFromText("Комфорт", текст_маркетолога, model="claude-sonnet-5")` ⇒ Создаётся черновик версии `(TariffId="comfort", version=1)`. AI строит `FormulaAST`:

```
base = quantity * 3.20    
overage = max(quantity - 300, 0) * 3.20 * 0.15    
total = base + overage
```

и `Predicate`: `applies_when: tariff_plan == "comfort"`. `SourceText` фиксирует исходный текст + версию модели-формализатора (провенанс).

1. `TariffVersion.Validate` ⇒ Домен проверяет:

   - все `reads` в `RuleBinding` резолвятся (например, если правило ссылается на `ReferenceParameter("vat_rate", "RU")`, параметр должен существовать);

   - AST не содержит противоречащих друг другу правил (например, два правила с пересекающимися `applies_when` и разными формулами для одной и той же величины);

   - AST синтаксически корректен и вычислим на whitelisted-DSL.

Если AI извлёк из текста двусмысленность (например, не указано, включает ли «свыше 300» само значение 300), валидация должна либо потребовать уточнения у маркетолога, либо зафиксировать выбранную интерпретацию явно в `SourceText`.

1. `TariffVersion.Publish` ⇒ Версия становится неизменяемой. С этого момента `BillingAssessment` может на неё ссылаться.

### События

`TariffDrafted` → `TariffValidated` → `TariffVersionPublished`.

### Что защищает агрегат

Единая транзакция гарантирует, что наружу не может «утечь» частично сформулированный или внутренне противоречивый тариф — расчёты по нему просто не начнутся, пока не пройден полный цикл Draft→Validate→Publish.

### Граничный случай

Если через месяц маркетолог решит изменить надбавку с 15% до 20% — это **не** правка `version=1`, а `DraftFromText` новой версии `version=2` с новым `TemporalValidity`. Старые `BillingAssessment`, посчитанные по `version=1`, не меняются задним числом — это отдельный сценарий (UC-7 работает только для `ReferenceParameter`, а изменение самого тарифа маркетингом задним числом в данной модели не предусмотрено намеренно).

## UC-3. Приём потребления

**Контекст.** Счётчик воды передаёт показание: аккаунт `acc-4471`, метрика `water_m3`, значение `12.4`, внешний id события `meter-778812-20260706`.

**Затронутые агрегаты:** только `ConsumptionStream`.

### Поток

`ConsumptionStream.RecordUsage(AccountId="acc-4471", metric="water_m3", quantity=12.4, external_event_id="meter-778812-20260706")` ⇒

Проверка идемпотентности: если событие с таким `ExternalEventId` уже записано — команда молча (или с явным сигналом «дубликат проигнорирован») завершается без повторной записи. Это критично, потому что счётчики и IoT-шлюзы регулярно шлют повторы при обрыве связи.

### События

`UsageRecorded`.

### Что защищает агрегат

Единственный инвариант — уникальность по внешнему id (FR-14). Никакой расчётной логики здесь нет: `ConsumptionStream` не знает про тарифы и не решает, сколько это будет стоить — это работа `BillingAssessment`, которая свернёт поток событий позже.

### Пример конфликта, который агрегат обязан отловить

Два шлюза одновременно шлют одно и то же показание с разными `external_event_id`, потому что диспетчер вручную продублировал заявку — это уже не дубликат с точки зрения агрегата (id разные), и оба факта будут записаны. Разрешение подобных семантических дублей — не забота `ConsumptionStream`, а либо `meta`-разметка (QA-оценка), либо отдельный процесс сверки выше по потоку.

## UC-4. Квитанция (on-timer)

**Контекст.** Наступило 1-е число месяца — по расписанию нужно выставить квитанцию аккаунту `acc-4471` за июнь 2026 по тарифу «Комфорт».

**Затронутые агрегаты:** `BillingAssessment` → `Invoice` → `Account`.

### Поток

1. `BillingAssessment.Calculate(AccountId="acc-4471", period="2026-06")` ⇒

   - Читает `TariffVersion("comfort", version=1)` — **по ссылке**.

   - Читает актуальные на конец периода значения `ReferenceParameter("vat_rate", "RU")` — например, версия действует с 2024-01-01, tx_to=∞.

   - Сворачивает все `UsageEvent` из `ConsumptionStream(acc-4471, "electricity_kwh")` за июнь: допустим, суммарно 340 кВт·ч.

   - Строит `ChargeLine`: база 300×3.20=960₽, превышение 40×3.20×1.15=147.2₽, итого 1107.2₽ до НДС.

   - Фиксирует `CalcContext`: версии правил, версии референсных констант, снапшот входного потребления, применённый AST, `steps` (трейс: quantity=340 → base=960 → overage=147.2 → total=1107.2 → с НДС=...).

   - Событие: `AssessmentCalculated`.

2. `Invoice.Issue(assessment_version)` ⇒

   - Замораживает `InvoiceLine` как **копию** `ChargeLine` (не ссылку) — квитанция от 1 июля не должна измениться, даже если завтра `BillingAssessment` будет пересчитан.

   - Событие: `InvoiceIssued`.

3. `Account.PostCharge(invoice_id, amount=1107.2)` ⇒

   - Append-only проводка в журнал `acc-4471`.

   - Событие: `EntryPosted`.

### Ключевой момент

Это **три отдельные транзакции**, связанные событиями через процесс-менеджер — не одна большая транзакция. Если на шаге 3 произошёл сбой, `Invoice` уже выставлен и неизменяем; повторная попытка проводки — отдельная операция восстановления (retry с идемпотентностью на уровне саги), а не откат всей цепочки.

## UC-5. Предпросмотр баланса (on-call)

**Контекст.** Пользователь открыл приложение 15 июня и хочет видеть «сколько я уже накопил в этом месяце», хотя официальная квитанция будет только 1 июля.

**Затронутые агрегаты:** `BillingAssessment` (preview) → `Account.ReservePending`.

### Поток

1. `BillingAssessment.Calculate(AccountId="acc-4471", period="2026-06", mode=preview)` ⇒ Считает по факту накопленного потребления на 15 июня (180 кВт·ч) — это **не** финальная версия периода, а предварительная оценка. Результат не порождает `Invoice`.

2. `Account.ReservePending(amount≈576₽)` ⇒ Создаёт `LedgerEntry` с `EntryType=pending`. Это видно пользователю как «ожидаемый расход», но не входит в подтверждённый баланс.

### Отличие от UC-4

Здесь нет `Invoice` вообще — предпросмотр не выставляет счёт. `Account` при этом обязан различать pending и posted проводки (two-phase), иначе пользователь увидит задвоенную сумму, когда 1 июля придёт финальная `posted`-проводка по тому же периоду. Двухфазность — это именно тот инвариант `Account`, который отделяет UC-5 от UC-4.

## UC-6. Перерасчёт по SLA/аварии (on-calc)

**Контекст.** 20 июня произошла авария на линии, и поставщик услуги задним числом (по акту от 25 июня) снижает начисление за 5 дней простоя — но правила тарифа и референсные константы **не менялись**, изменились только входные данные (пересмотренное потребление или признак простоя в `meta`).

**Затронутые агрегаты:** `BillingAssessment.Recalculate` → `Invoice.IssueCorrecting` → `Account.PostCorrection`.

### Поток

1. `BillingAssessment.Recalculate(AccountId="acc-4471", period="2026-06")` ⇒ Те же версии правил (`comfort v1`) и те же версии референсных констант, но новые входные данные (пересмотренное потребление с учётом простоя, например 340 → 310 кВт·ч). Старая версия `AssessmentCalculated` помечается superseded, атомарно создаётся новая. Событие `AssessmentRecalculated` несёт **diff**: `{quantity: 340→310, total: 1107.2→991.2}`.

2. `AssessmentRecalculated` → (сага) → `Invoice.IssueCorrecting(original_invoice_id)` ⇒ Новая квитанция с `CorrectionLink` на исходную. Исходная квитанция от 1 июля **не удаляется и не мутирует** — она остаётся историческим фактом.

3. `Invoice.CorrectingInvoiceIssued` → (сага) → `Account.PostCorrection(delta=-116₽)` ⇒ Проводка-корректировка, а не правка старой проводки — журнал остаётся append-only.

### Почему это отдельный от UC-7 юзкейс

Здесь пересчёт триггерится **бизнес-событием** (авария, акт от поставщика), а не изменением закона. Формально поток команд идентичен UC-7, но источник инициирующего события другой: в UC-6 инициатор — оператор/саппорт, в UC-7 — `ReferenceParameterCorrected`.

## UC-7. Перерасчёт по закону

**Контекст.** 10 июля 2026 региональный регулятор публикует нормативный акт: НДС для коммунальных услуг задним числом за июнь снижен с 20% до 10% (льгота, введённая законом от 5 июля, действующим с 1 июня 2026). Это ретроактивная коррекция **valid-time** — норма на самом деле действовала иначе, чем система полагала до сих пор.

**Затронутые агрегаты:** `ReferenceParameter.Correct` → `BillingAssessment.Recalculate` (для **всех** затронутых аккаунтов) → `Invoice.IssueCorrecting` → `Account.PostCorrection`.

### Поток

1. `ReferenceParameter.Correct(key="vat_rate", jurisdiction="RU", new_value=0.10, valid_from="2026-06-01", provenance=Ссылка_на_акт)` ⇒

   - Старая версия `ParameterValueVersion(0.20)` имела `valid_time=[2024-01-01, ∞)` — заявление о том, когда норма **действует по закону** — и `tx_time=[дата_регистрации, ∞)` — период, когда система держала это значение как актуальное.

   - Команда **не удаляет** старую версию — она закрывает её `tx_to` на `2026-07-10` (момент, когда система *перестала считать* значение 0.20 актуальным представлением закона). `valid_time` старой версии при этом не переписывается — она как была `[2024-01-01, ∞)`, так и остаётся: это исторический факт о том, что раньше система *утверждала*, что норма действует именно так.

   - Создаётся новая версия: `value=0.10`, `valid_time=[2026-06-01, ∞)` (когда норма *действительно* вступила в силу по закону), `tx_time=[2026-07-10, ∞)` (когда система впервые об этом узнала и зафиксировала).

   - Проверяется инвариант: среди **актуальных** (`tx_to=∞`) версий нет пересечений по `valid_time`. Старая версия больше не актуальна (`tx_to` закрыт), поэтому пересечения `valid_time=[2024-01-01,∞)` со старой версией и `valid_time=[2026-06-01,∞)` с новой не нарушают инвариант.

   - Событие: `ReferenceParameterCorrected`.

2. `ReferenceParameterCorrected` → (процесс-менеджер) находит **все** `BillingAssessment`, чей период пересекает скорректированный `valid_time` (то есть все периоды с 1 июня 2026 по всем аккаунтам, тарифы которых читают `vat_rate/RU`) ⇒ Для каждого из, скажем, 40 000 затронутых аккаунтов: `BillingAssessment.Recalculate` — с **новой** версией референсной константы, но с теми же `TariffVersion`. Каждый пересчёт — независимая транзакция, поэтому массовая операция параллелится по инстансам агрегата.

3. Далее для каждого аккаунта — та же цепочка, что в UC-6: `Recalculate` → `IssueCorrecting` → `PostCorrection`.

### Ключевое отличие от UC-6

Триггер — не бизнес-факт по одному аккаунту, а изменение общего для всех разделяемого `ReferenceParameter`. Отсюда и масштаб: сага здесь управляет не одним пересчётом, а массовым веером пересчётов по всем аккаунтам, чей период пересекается с изменённым `valid_time`.

### Что показывает `as-of tx`-запрос (польза от разделения осей)

Запрос *«что система считала действующим для даты 15 июня 2026, если бы мы спросили об этом 5 июля 2026 (до коррекции)?»* вернёт **0.20** — потому что на момент `tx=5 июля` старая версия ещё не была закрыта (`tx_to` ещё `∞`). Тот же запрос, заданный сегодня (после 10 июля), вернёт **0.10**. Это и есть смысл `tx_time` — «что мы знали в тот момент», в отличие от `valid_time`, которая отвечает только на вопрос «когда норма действовала по закону».

### Что было бы нарушено без битемпоральности

Если бы `ReferenceParameter` хранил только `valid_time` (без `tx_time`), после коррекции было бы невозможно объяснить в UC-9, почему квитанция, выставленная 1 июля, считалась по ставке 20%, а перевыставленная 15 июля — по 10%: история «во что система верила в момент расчёта» была бы физически потеряна, а не просто скрыта.


## UC-8. Пересмотр оценки QA-агента

**Контекст.** AI-агент QA пересматривает автоматически выставленную оценку по тикету поддержки: изначально было зафиксировано «5 часов простоя сервиса», после ручной проверки супервайзером — «на самом деле 3 часа, остальное — некорректная фиксация агента».

**Затронутые агрегаты:** `ConsumptionStream.RecordUsage` (как новый факт) → `BillingAssessment.Recalculate` → далее как UC-6 (`Invoice.IssueCorrecting` → `Account.PostCorrection`).

### Поток

1. `ConsumptionStream.RecordUsage(AccountId="acc-9012", metric="downtime_hours", quantity=3, external_event_id="qa-review-77213", meta={reviewed_by: "supervisor-42", supersedes_meta: "qa-agent-v2-run-441"})` ⇒ Важно: это **новое** событие потребления, а не правка старого `UsageEvent` — `ConsumptionStream` append-only, пересмотр оценки не мутирует исходную запись агента, а добавляет корректирующий факт с провенансом «кто и почему пересмотрел».

2. `UsageRecorded` → (сага) → `BillingAssessment.Recalculate` — сворачивает поток заново, теперь с учётом пересмотренного факта (3 часа вместо 5).

3. Далее идентично UC-6: `Invoice.IssueCorrecting` → `Account.PostCorrection`.

### Отличие от UC-6 и UC-7

Источник изменения — не бизнес-акт по аварии и не закон, а пересмотр **входных данных потребления** конкретно этим механизмом контроля качества AI-агента. С точки зрения `BillingAssessment` это неотличимо от UC-6 (те же версии правил, новые данные) — специфика UC-8 целиком живёт на уровне `ConsumptionStream` и провенанса в `meta`.

## UC-9. Объяснение «за что я плачу?»

**Контекст.** Пользователь `acc-4471` в приложении жмёт «Почему в этом месяце так дорого?» по квитанции за июнь.

**Затронутые агрегаты:** read-only query над `BillingAssessment.CalcContext` (+ провенанс `ReferenceParameter`). **Не команда**, инвариант ничем не рискует.

### Пример разворачиваемого объяснения

Запрос читает `CalcContext` зафиксированной версии `BillingAssessment(acc-4471, 2026-06)`:

- Применённый тариф: `comfort v1`, действует с 2026-07-01 (текст-источник и модель-формализатор доступны через `SourceText`).

- Снапшот потребления: 340 кВт·ч (ссылка на конкретные `UsageEvent` через `ConsumptionStream`).

- `steps` (трейс): `база 300×3.20=960 → превышение 40×3.20×1.15=147.2 → сумма=1107.2 → НДС 20% (версия ставки, действовавшая на момент расчёта, с провенансом на акт) → итого`.

- Провенанс ставки НДС: ссылка на `ReferenceParameter("vat_rate","RU")`, конкретная `ParameterValueVersion`, и в ней — `Provenance` (номер нормативного акта, дата вступления).

### Почему это в принципе возможно

Возможность ответить на этот вопрос — прямое следствие того, что `CalcContext` пиннит **версии**, а не значения: даже если сегодня ставка НДС уже 10% (после UC-7), объяснение по июньской квитанции **до** коррекции честно покажет «на момент расчёта считали 20%», а не подставит текущее значение задним числом.

## UC-10. Diff версий квитанции

**Контекст.** После UC-7 (льгота по НДС) пользователь видит, что квитанция за июнь изменилась, и хочет понять — что именно.

**Затронутые агрегаты:** query `BillingAssessment.diff(v1, v2)` — **не команда**, чистое сравнение двух уже существующих версий одного и того же корня.

### Пример вывода diff

```
BillingAssessment(acc-4471, 2026-06)    
  v1 (2026-07-01, superseded)         v2 (2026-07-10, актуальная)    
  ─────────────────────────────────   ─────────────────────────────────    
  vat_rate: 0.20 (версия A)     →     vat_rate: 0.10 (версия B)    
  base: 960.00                        base: 960.00           (не изменилось)    
  overage: 147.20                     overage: 147.20        (не изменилось)    
  subtotal: 1107.20                   subtotal: 1107.20      (не изменилось)    
  vat_amount: 221.44             →    vat_amount: 110.72    
  total: 1328.64                 →    total: 1218.56    
  причина пересчёта: ReferenceParameterCorrected (акт от 2026-07-05)
```

### Ключевое свойство

Diff — не отдельное состояние, а вычисление над двумя `CalcContext`, которые уже честно хранятся как versioned snapshots внутри одного агрегата `BillingAssessment`. Именно поэтому в карте агрегатов diff **осознанно не агрегат**: заводить для него отдельный корень означало бы дублировать состояние, которое уже полностью выводимо из существующих версий.
