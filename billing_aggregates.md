# Доменные агрегаты AI-native биллинга

**Статус:** черновик v0.1 (концепт-фаза)
**Стек хранения:** PostgreSQL (битемпоральность реализуется вручную — range-типы + exclusion-constraints).

---

## Принцип, по которому проведены границы

Агрегат — это **граница инварианта, который должен держаться консистентным в одной транзакции немедленно**. Всё, что можно согласовать отложенно (eventual), вынесено в отдельный агрегат и связано доменными событиями.

Два ключевых следствия:
- **Одна транзакция меняет один агрегат.** Между агрегатами — событийная (eventual) согласованность через процесс-менеджер (сагу).
- Сущность имеет id и жизненный цикл внутри агрегата; на неё **нельзя ссылаться снаружи** — только через корень. Value Object неизменяем и не имеет идентичности.

---

## Карта агрегатов

| Агрегат | Идентичность | За какой инвариант отвечает |
|---|---|---|
| **TariffVersion** | (TariffId, version) | Правила внутри версии непротиворечивы и реализуемы |
| **ReferenceParameter** | (key, jurisdiction) | Нормы/налоги не пересекаются по valid-time; есть провенанс |
| **BillingAssessment** | (AccountId, period) | Строки расчёта за период консистентны, без задвоения, каскады когерентны |
| **Invoice** | InvoiceId | Выставленная квитанция неизменяема |
| **Account** | AccountId | Двойная запись; баланс = свёртка неизменяемых проводок |
| **ConsumptionStream** | (AccountId, metric) | Идемпотентность приёма событий потребления |

 │ReferenceParameter││                  │              └──────────┘             └──────────┘

## 1. TariffVersion

Формализованный тариф. Результат AI-формализации человеческого описания от маркетолога/регулятора.

- **Идентичность:** `Tariff` — нить версий (`TariffId`); граница консистентности — **одна версия** `(TariffId, version)`, т.к. правила должны быть непротиворечивы между собой в пределах версии, а разные версии независимы.
- **Сущности (entities):**
  - `TariffRule` — правило: условие применимости + формула. Имеет id внутри версии.
- **Value Objects:**
  - `CatalaSource` — исходный текст на Catala (литературный блок: цитата нормы + код). Хранится всегда, а не «на всякий случай» — это недостающее звено между `SourceText` (человеческий текст + модель-формализатор) и непрозрачным скомпилированным артефактом. Без него нельзя объяснить, почему из абзаца договора получилась именно эта цифра.
  - `ScopeManifest` — явный контракт скоупа. Именно он делает `RuleBinding.reads/produces` проверяемым инвариантом: скомпилированный артефакт сам по себе не сообщает рантайму, какие `ReferenceParameter` ему нужны.

    ```
    ScopeManifest:
      scope_name : str                     -- точка входа
      inputs  : [ (arg_name, type, binding) ]
                binding ∈ { RefParam(key, jurisdiction)
                          | Metric(name)
                          | AccountState(field) }
      outputs : [ (arg_name, type → ChargeLine | State) ]
    ```

  - `ArtifactRef` — координаты и идентичность скомпилированного модуля: `source_hash`, `compiler_version`, `runtime_version`.
  - `Predicate` — условие применимости (`applies_when`) как AST-предикат.
  - `RuleBinding` — что правило читает (`reads`) и какое начисление производит (`produces`).
  - `TemporalValidity` — период действия версии.
  - `SourceText` — исходный человеческий текст + версия модели, которая формализовала (провенанс формализации).

  **Идентичность артефакта** — пара `(sha256(CatalaSource), toolchain_version)`, где `toolchain_version` = связка версии компилятора и версии рантайм-библиотеки Catala. Хеш берётся **от исходника Catala, а не от сгенерированного Python**: кодогенерация может быть недетерминированной в мелочах (порядок объявлений, метаданные), и хеш от Python давал бы разные значения при идентичной семантике.

  `runtime_version` фиксируется отдельно от `compiler_version`, хотя на практике это одна связка: сгенерированный модуль импортирует рантайм Catala (собственные типы денег, дат, длительностей; обработка конфликта дефолтов) и рассчитан на конкретную его версию.
- **Инварианты:** все `RefParam`-binding'и из `ScopeManifest` резолвятся в реестре `ReferenceParameter`; `CatalaSource` компилируется без ошибок (`clerk build`); нет взаимно противоречащих правил (перекрывающихся `applies_when` для одной величины — см. UC-1/2); публикуется только целиком провалидированной; после `Publish` неизменяема.
- **Команды:**
  - `DraftFromText` — AI формализует человеческий текст в `CatalaSource` + `ScopeManifest`.
  - `Validate` — здесь происходит **сборка**: `clerk build` (компиляция), проверка, что все `RefParam`-binding'и из `ScopeManifest` резолвятся в реестре `ReferenceParameter`, прогон golden-тестов, опционально `catala proof`. Результат — собранный артефакт лежит в реестре `tariff_artifact` в статусе «провалидирован».
  - `Publish` — помечает уже собранный артефакт активным. Остаётся честной однотранзакционной командой над одним агрегатом.

  Сборка происходит на `Validate`, а не на `Publish`, сознательно: компиляция может упасть, и падать она должна там, где это ошибка формализации, а не в момент публикации. Иначе `Publish` перестаёт быть атомарной командой над агрегатом и требует промежуточного статуса `Compiling`.
- **События:** `TariffDrafted`, `TariffValidated`, `TariffVersionPublished`.

## 2. ReferenceParameter

Внешний референсный контекст расчёта: налоги, нормы ЖКХ, законодательные коэффициенты. Самый чистый носитель битемпоральности. **Не принадлежит никакому аккаунту** — общий, расчёт ссылается на него по (ключ, юрисдикция, id-версии).

- **Идентичность:** `(key, jurisdiction)` — например `("vat_rate", "RU")`. Граница выбрана так, чтобы инвариант непересечения был проверяем в одной транзакции, и не шире.
- **Сущности:**
  - `ParameterValueVersion` — версия значения; занимает прямоугольник в двумерном времени (valid-time × transaction-time). На неё пиннится `CalcContext`.
- **Value Objects:**
  - `ParameterValue` — **полиморфное** значение: скаляр (`0.20`), пороговая таблица (соцнорма), прогрессивная шкала. Тип (`kind`) стабилен в пределах параметра.
  - `TemporalValidity` — valid-time: когда норма действует **по закону**.
  - `TransactionTime` — tx-from/tx-to: когда система **считала это правдой**.
  - `Provenance` — первоклассный VO: ссылка на нормативный акт + id текста в document store + дата вступления. Без него обрывается объяснимость.
- **Инварианты:** нет пересечений valid-time среди **актуальных** версий (`tx_to = ∞`); **пропуски допустимы, пересечения — нет**; провенанс обязателен; история физически неизменяема (коррекция закрывает `tx_to`, не удаляет).
- **Команды:** `RegisterValue` (обычная новая норма), `Correct` (ретроактивная коррекция убеждения — UC-7), `Repeal` (отмена нормы с даты — «удаление» через INSERT, обрезающий valid-to).
- **События:** `ReferenceParameterRegistered`, `ReferenceParameterCorrected`, `ReferenceParameterRepealed`.
- **Не входит:** текст закона (в document store), резолвинг иерархии юрисдикций (доменный сервис снаружи), «сплющенный» ruleset (read-model).

## 3. BillingAssessment

Сердце домена — расчёт для **одного аккаунта за один период**. Инстансов миллионы, они независимы, поэтому массовый пересчёт параллелится.

- **Идентичность:** `(AccountId, BillingPeriod)` — нить; версии внутри (каждый пересчёт = новая версия, старая помечается superseded).
- **Сущности:**
  - `ChargeLine` — начисление (строка расчёта): какое правило сработало, за какой период услуги, сумма. Имеет id, но **снаружи адресуется только через корень**.
- **Value Objects:**
  - `CalcContext` — снапшот контекста расчёта, несущая конструкция воспроизводимости и объяснимости. Пиннит:
    - версии применённых правил (`TariffId`, version, rule id);
    - **версии референсных констант** (`key`, `jurisdiction`, id-версии-значения) — не значения, а ссылки на версии;
    - снапшот входного потребления;
    - ссылку на артефакт формулы: `artifact_hash` (= `source_hash` артефакта Catala) и `toolchain_version` — без них воспроизведение расчёта не гарантирует тот же результат (см. UC-9/UC-10);
    - `steps` — трейс промежуточных вычислений (для UC-9; на массовом пути не материализуется, см. UC-9).
  - `Money` — сумма + валюта + правило округления.
  - `Quantity` — значение + метрика.
- **Инварианты:** активные (не-superseded) строки внутри аккаунта+периода консистентны — нет задвоения; каскадные правила (скидка за объём зависит от суммарного потребления) применены когерентно; `Recalculate` атомарно помечает прежнюю версию superseded и создаёт новую. **Именно этот инвариант «через несколько строк сразу» обосновывает единый агрегат, а не отдельные `Charge`.**
- **Команды:** `Calculate` (первая версия), `Recalculate` (новая версия).
- **События:** `AssessmentCalculated`, `AssessmentRecalculated` (несёт diff).

## 4. Invoice

Квитанция — зафиксированный на момент выставления снапшот начислений.

- **Идентичность:** `InvoiceId`.
- **Сущности:**
  - `InvoiceLine` — замороженная **копия** строки начисления (не ссылка — копия, чтобы квитанция не менялась вслед за расчётом).
- **Value Objects:**
  - `Money` в каждой строке и итог.
  - `CorrectionLink` — ссылка на исходную квитанцию (для корректирующих).
- **Инварианты:** выставленная квитанция **никогда не мутирует**; корректировка порождает новую квитанцию, связанную с прежней.
- **Команды:** `Issue` (заморозить версию assessment), `IssueCorrecting` (после перерасчёта → новая, ссылается на исходную).
- **События:** `InvoiceIssued`, `CorrectingInvoiceIssued`.

---

## 5. Account (лицевой счёт)

Деньги. Журнал проводок (append-only), баланс — производная.

- **Идентичность:** `AccountId`.
- **Сущности:**
  - `LedgerEntry` — проводка. Неизменяема после записи.
- **Value Objects:**
  - `Money` — сумма проводки.
  - `EntryType` — pending / posted (two-phase).
  - `CorrectionLink` — связь корректирующей проводки с исходной.
- **Инварианты:** двойная запись; проводки неизменяемы; баланс = сумма проводок (не хранимое поле); two-phase (pending → posted).
- **Команды:** `ReservePending` (on-call прикидка), `ConfirmPending` (on-timer фиксация), `PostCharge`, `PostCorrection`.
- **События:** `PendingReserved`, `EntryPosted`, `CorrectionPosted`.

---

## 6. ConsumptionStream (тонкий агрегат)

Событие потребления неизменяемо, защищать почти нечего — единственный реальный инвариант — идемпотентность приёма. Само событие правильнее воспринимать как входящее доменное событие, а не полноценный агрегат.

- **Идентичность:** `(AccountId, metric)`.
- **Сущности:** `UsageEvent` — факт потребления (12.4 м³ / закрытая заявка / МБ).
- **Value Objects:** `Quantity` (значение + метрика), `ExternalEventId` (ключ идемпотентности), `meta` (QA-оценка, SLA-контекст).
- **Инвариант:** уникальность события по внешнему id — один факт не записывается дважды (FR-14).
- **Команда:** `RecordUsage` (идемпотентная).
- **Событие:** `UsageRecorded`.

## Общие Value Objects

Переиспользуются между агрегатами, несут инварианты:

- **`Money`** — сумма + валюта + правило округления (округление юридически значимо).
- **`Quantity`** — значение + метрика (общая абстракция над разнородным потреблением).
- **`TemporalValidity`** — полуоткрытый интервал valid-time (`valid_to = null` → «до отмены»).
- **`CalcContext`** — запиненный снапшот всего, что использовал расчёт, включая `artifact_hash` и `toolchain_version` (см. `BillingAssessment`).
- **`Provenance`** — прослеживаемость до первоисточника.

---

## Что осознанно НЕ агрегат

- **Перерасчёт** — не сущность, а *команда* `Recalculate` над `BillingAssessment` + событие с diff. Отдельный агрегат был бы анемичной обёрткой над процессом.
- **Diff** — *query* над двумя версиями одного `BillingAssessment`, не состояние.
- **ChargeLine / InvoiceLine / LedgerEntry** — сущности внутри своих корней, не агрегаты.
- **«Сплющенный» ruleset на период** — read-model/проекция для быстрого on-call, а не состояние.

## Реестр артефактов и порт `FormulaEngine`

**Реестр `tariff_artifact`** — не агрегат, а инфраструктурная таблица. Записи неизменяемы после `Publish`, их немного (одна опубликованная версия тарифа = одна запись).

```
tariff_artifact
  tariff_id, version        -- PK; те же координаты, что пиннит CalcContext
  catala_source             -- text
  source_hash               -- sha256(catala_source)
  compiler_version
  runtime_version
  scope_name
  scope_manifest            -- jsonb
  compiled_py               -- bytea | ссылка на объектное хранилище
  built_at
```

**Порт `FormulaEngine`** — интерфейс в домене, реализация снаружи:

```
execute(artifact_ref, CalcInput) → (ChargeLines, steps) | ConflictError
```

За портом сидит всё, что специфично для Catala: загрузка модуля по `artifact_ref`, кеширование по `(tariff_id, version)` в процессе воркера, конверсия типов `ParameterValue → типы рантайма Catala` и обратно `результат → Money`, отлов ошибки конфликта дефолтов.

Конверсия типов обязана жить **в одном месте** (в реализации порта), потому что правило округления юридически значимо (см. VO `Money`) и не должно дублироваться одновременно и в `Money`, и в конверторе.

**Агрегат `BillingAssessment` не знает, что за портом стоит Catala.** Он получает строки начисления и проверяет свои инварианты. Если движок когда-нибудь заменят — домен не шелохнётся.

## Резолвинг референсных параметров

Между `ReferenceParameter` (битемпоральный, с историей версий) и входом скоупа (`decimal`, одно число) есть зазор. Его закрывает доменный сервис — по той же логике, по которой уже вынесен наружу резолвинг иерархии юрисдикций:

```
resolve(key, jurisdiction, valid_on, as_of_tx) → (value, version_id)
```

- В скоуп уходит `value`. Catala про версии параметров не знает и знать не должна — она работает на одном снапшоте.
- В `CalcContext` уходит `version_id`. Это разные пути, и терять второй нельзя — на нём держится объяснимость (UC-9).

Значения аргументов по сценариям:

| Сценарий | `valid_on` | `as_of_tx` |
|---|---|---|
| `Calculate` (UC-4/5) | конец расчётного периода | `now` |
| `Recalculate` (UC-6/7/8) | конец расчётного периода | `now` |
| Воспроизведение прежней версии расчёта (UC-9/10) | конец расчётного периода | момент того расчёта |

Две ошибки, которые эта сигнатура предотвращает:

1. **`valid_on = now` вместо конца периода.** Пересчитывая июнь пятого июля, надо взять ставку, действовавшую **в июне**. Иначе смена нормы 1 июля отравляет пересчёт июня.
2. **`as_of_tx` всегда `∞`.** Тогда повторный прогон версии `v1` в UC-10 даст сегодняшнее значение, и diff схлопнется: показать «`vat_rate`: 0.20 → 0.10» станет невозможно. Это единственный случай, где `as_of_tx ≠ now`, и ровно ради него в `ReferenceParameter` живёт вторая ось.

### Кто выполняет резолвинг

**Не агрегат.** `BillingAssessment.Calculate` защищает инвариант когерентности строк; он не должен знать про реестр артефактов, `importlib` и битемпоральные запросы. Резолвинг параметров и загрузка артефакта — прикладной слой.

Порядок в прикладном слое (вызывается сагой):

1. Резолв активной `(TariffId, version)` для аккаунта на период.
2. Чтение `ScopeManifest` → список требуемых `RefParam`-ключей.
3. `resolve(...)` для каждого → `(value, version_id)`.
4. Свёртка `ConsumptionStream` за период → снапшот потребления.
5. Сборка `CalcInput` = значения параметров + снапшот + состояние аккаунта.
6. Сборка заготовки `CalcContext` = `(TariffId, version)`, `artifact_hash`, `toolchain_version`, список `version_id`, снапшот входов.
7. `BillingAssessment.Calculate(CalcInput, CalcContext, FormulaEngine)`.

Практическое обоснование, а не только чистота слоёв: шаги 1–3 **общие для всех аккаунтов на одном тарифе и периоде**. В веере UC-7 на 40 000 аккаунтов резолвинг делается один раз на пачку. Если бы он сидел внутри агрегата, вынести его было бы нельзя.

## Связи между агрегатами

- `BillingAssessment` **читает** `TariffVersion` (по TariffId+version) и значения `ReferenceParameter` (по key+jurisdiction+id-версии) — **по ссылке, не владеет** ими.
- `BillingAssessment` **сворачивает** события из `ConsumptionStream`.
- `Invoice` **замораживает** конкретную версию `BillingAssessment`.
- `Account` **проводит** начисления из `Invoice`.
- **Саги (eventual, через события):**
  - `ReferenceParameterCorrected` → процесс находит затронутые `BillingAssessment` (чей период пересекает скорректированный valid-time) → `Recalculate`.
  - `AssessmentRecalculated` → `Invoice.IssueCorrecting` → `Account.PostCorrection`.

Ни один из этих переходов не идёт в одной транзакции — только через доменные события и процесс-менеджер.

## Участие агрегатов в юзкейсах

Формат: команда над агрегатом. Стрелка `→` внутри строки = переход через доменное событие (eventual).

| Юзкейс | Поток команд по агрегатам |
|---|---|
| **UC-1/2** Постановка и валидация тарифа | `TariffVersion.DraftFromText` → `TariffVersion.Validate` → `TariffVersion.Publish` |
| **UC-3** Приём потребления | `ConsumptionStream.RecordUsage` (идемпотентно) |
| **UC-4** Квитанция (on-timer) | `BillingAssessment.Calculate` → `Invoice.Issue` → `Account.PostCharge` |
| **UC-5** Предпросмотр баланса (on-call) | `BillingAssessment.Calculate` (preview) → `Account.ReservePending` |
| **UC-6** Перерасчёт SLA/авария (on-calc) | `BillingAssessment.Recalculate` (те же версии правил, новые данные) → `Invoice.IssueCorrecting` → `Account.PostCorrection` |
| **UC-7** Перерасчёт по закону | `ReferenceParameter.Correct` → `BillingAssessment.Recalculate` (новые версии констант) → `Invoice.IssueCorrecting` → `Account.PostCorrection` |
| **UC-8** Пересмотр оценки QA агента | `ConsumptionStream.RecordUsage` (пересмотр как новый факт) → `BillingAssessment.Recalculate` → далее как UC-6 |
| **UC-9** Объяснение «за что я плачу?» | read-only query над `BillingAssessment.CalcContext` (+ провенанс `ReferenceParameter`) — не команда |
| **UC-10** Diff версий квитанции | query `BillingAssessment.diff(v1, v2)` — не команда |

## Ключевой вывод

Почти каждый расчётный юзкейс (UC-4/6/7/8) пересекает границы агрегатов по цепочке **Assessment → Invoice → Account**. Поэтому согласованность между ними — **eventual, через события и саги**, а строгая немедленная консистентность живёт **внутри** каждого агрегата. Границы проведены именно так, чтобы денежный инвариант целиком лежал внутри `Account`, а каскадный расчётный — целиком внутри `BillingAssessment`.
