# Формализация «Claude Family Pro» — пояснительная записка

Формализация вычислимых пунктов оферты `Case.md` (версия 3.2, юрисдикция RU) на
Catala 1.2.0 с двумя контурами проверки: быстрый — `#[test]`-скоупы Catala, и
приёмочный — `pytest` поверх **транспилированного Python** (именно этот код едет в
рантайм).

## Файлы

| Файл | Назначение |
|------|-----------|
| `claude_family_pro.catala_en` | Модель: структуры, константа-провенанс `const_tariff`, скоупы правил, композитный `GroupPeriodBill`. |
| `claude_family_pro_tests.catala_en` | 36 `#[test]`-скоупов, фиксирующих ожидаемые выходы через `assertion`. |
| `test_claude_family_pro.py` | 41 `pytest` поверх сгенерированного Python-модуля; входы рантайма замоканы. |
| `build_pyrun.sh` | Собирает импортируемый Python-пакет (модель + транспилированный stdlib + рантайм) в `_build/pyrun/cfp`. |
| `Makefile` | Цели `setup / typecheck / test / python / python-pkg / pytest / clean`. |

## Как запускать

```
make typecheck   # Typechecking successful!
make test        # Catala #[test]-скоупы, exit 0 при успехе (123 при провале assertion)
make python      # обязательный артефакт: _build/python/Claude_family_pro.py
make pytest      # сборка пакета + 41 pytest поверх транспилированного Python
```

Тулчейн — opam-switch `catala` (Catala 1.2.0). `make test` использует
`catala interpret --whole-program`, потому что `clerk test` в этой раскладке не
линкует кросс-файловый модуль (см. заметку по тулчейну).

## Входы, которые заполняет рантайм (не Catala)

Catala формализует правила, но не хранит состояние между периодами и не владеет
внешними данными. Эти величины — `input` скоупов; рантайм подаёт их при вызове, а
в `pytest` они замоканы:

- **`carried_balance`** (НТ) — неизрасходованный включённый объём, перенесённый с
  прошлого периода (§4). Состояние рантайма: он хранит остаток, подаёт его на вход
  и обнуляет по §4.2 (сгорание). Модель только *потребляет* его и вычисляет новый
  `carryover_to_next`, но **не помнит** между запусками.
- **`consecutive_paid_periods`** — номер текущего периода в непрерывной серии
  оплаченных периодов (§5.4). Непрерывность и её обрыв приостановкой > 3 дней ведёт
  рантайм; модель лишь сравнивает с порогом 13.
- **`referral_active`** — действует ли в этом периоде реферальная скидка (§5.5);
  зависит от факта приглашения в прошлом периоде — учёт рантайма.
- **`availability`** (доля 0..1) — измеренная доступность за период (§7). Может
  утверждаться задним числом (§7.4) — это отдельная корректирующая строка рантайма.
- **`vat_rate`** — ставка НДС юрисдикции Владельца на последний день периода (§8).
  Нормативная, битемпорально версионируемая величина; §8.2/§9.2 — ретроактивные
  изменения — это подача рантаймом другого значения и перевыставление.
- **`is_termination`, `active_days`, `days_in_month`** — календарь периода и факт
  расторжения (§2.2). Владелец рантайма — биллинговый календарь.
- **`members`** — список Участников с уже **нормализованным** потреблением за
  период (нормализация — отдельный скоуп §1.2), флагами `is_child`/`is_owner`.

## Зафиксированные интерпретации двусмысленностей

Юридический текст неоднозначен; выбор зафиксирован в прозе модели, в коде и в
граничном тесте. Ключевые решения:

1. **§5.1 vs §5.3 (маржинальные диапазоны при fair use).** Fair-use-часть овеража
   (потребление Участника свыше 30 млн НТ) полностью вынута из скидочной базы и
   тарифицируется по полной ставке. Маржинальные диапазоны §5.1 отсчитываются
   **от нуля по скидочной части** (овераж минус fair-use), а не по полному оверажу.
   Тесты: `TestChargeFairUse` / `test_charge_fair_use_excluded`.
2. **§5.2 (порог лояльности).** Условие «≥ 20 млн» проверяется по **полному**
   оверажу Группы; единая скидка 25% применяется к **скидочной** части (fair-use
   по-прежнему исключён). Тест: `TestChargeLoyalty`, `TestChargeFairUse`.
3. **§7.1 (границы SLA).** Пороги трактуются строго: `< 99.5%` не включает 99.5%
   (на 0.995 компенсации нет); границы 98.0% и 95.0% попадают в верхний диапазон
   («≥ 98%», «≥ 95%»). Тест-параметризация `test_sla_compensation`.
4. **§2.2 / §6.2 (округление).** Пропорция абонплаты и доли Участников округляются
   **вниз** до копейки. Важно: `money * decimal` в Catala округляет к *ближайшей*
   копейке, поэтому флор применяется к decimal-произведению до перевода в деньги
   (иначе 2490·10/31 дало бы 803.23 вместо 803.22). Тесты:
   `TestAllowanceTerminationRounding`, `TestDistRemainder`.
5. **§6.1/§6.3 (знаменатель доли).** База разнесения — нормализованное потребление
   **всех** Участников, включая детские аккаунты; доля ребёнка вычисляется, но
   целиком относится на Владельца. Владелец также поглощает остаток округления.
   Тесты: `TestDistChild`, `TestDistClean`.
6. **§6 (какая сумма разносится).** Разносится пред-налоговое начисление `pretax`
   (после скидок, до НДС и компенсации), т.к. НДС — финальное действие (§8.1), а
   компенсация — отдельная строка, возможно задним числом (§7.4).
7. **§2.3 (минимум).** Пол «не меньше абонплаты» применяется к пред-налоговой сумме
   после скидок и **не** учитывает компенсацию (§7.3); при расторжении (§2.2) пол не
   действует. Тесты: `TestCapAndFloor`, `TestFloorNoOverage`, `TestNoFloorOnTermination`.

## Инварианты

- Овераж, перенос, налогооблагаемая база, возвраты — неотрицательны (`if <0 then 0`).
- Перенос ограничен 20% включённого объёма.
- Совокупная скидка ограничена 30% суммы (абонплата + овераж).
- Разнос сходится точно: сумма всех начислений = начислению Группы (проверяется в
  каждом тесте разнесения как `alloc_sum = group_charge`).

## Провенанс и переиспользование (§9.1)

Все ставки/пороги/коэффициенты — в одной структуре `Tariff` и одной константе
`const_tariff` (пиннятся на дату акцепта). Каждый скоуп берёт `context tariff` со
значением по умолчанию `const_tariff`, поэтому другой экземпляр договора
обслуживается тем же кодом переопределением `tariff` — это проверяет
`test_tariff_override_reuses_scope`.

## Прозаические (невычислимые) пункты — вне модели

§1.1 (состав Группы, роль Владельца как факт), §1.3 (определение периода как
календарной сущности — рантайм), §3.2/§5.x (кто именно тарифицируется — это уровень
Группы, отражён структурой, а не формулой), §7.4 (момент выставления компенсации),
§8.2/§9 (процедура перерасчёта и изменения условий) — записаны прозой как контекст,
но не как исполняемые формулы; вычислимые следствия §8.2/§9.2 сведены к подаче
рантаймом версионированных значений (`vat_rate`).

## Что проверяют тесты

Оба контура покрывают каждую ветку `if/then/else` модели и каждую зафиксированную
интерпретацию. Catala-тесты (`assertion`) — быстрый контур во время моделирования;
негативный путь подтверждён: намеренная порча одного `assertion` роняет прогон с
кодом 123. `pytest` — приёмочный контур: он вызывает **скоупы транспилированного
Python** напрямую, мокая входы рантайма (`carried_balance`, `vat_rate`,
`availability`, `consecutive_paid_periods`, `referral_active`, календарь
расторжения) конкретными значениями под граничные случаи, включая сценарии, которых
нет в Catala-контуре (`test_e2e_carried_balance_mocked_state`,
`test_final_retroactive_vat_change`, `test_tariff_override_reuses_scope`).



- TokenNormalization — сырые токены → НТ по коэффициентам §1.2;
- PeriodAllowance — абонплата и включённый объём, пропорция при расторжении;
- OverageComputation — списание «перенос → включённый → овераж», перенос ≤20%;
- OverageCharge — тарификация овеража + скидки §5.1–5.3 (шкала vs лояльность, вычет fair-use);
- LoyaltyReferralDiscount — стаж vs реферальная, max;
- DiscountCapAndTotal — потолок 30% + минимум §2.3;
- SLACompensation — компенсация по доступности;
- FinalBill — НДС и итог;
- ParticipantDistribution — разнос по долям, round-down, остаток+дети Владельцу;
- GroupPeriodBill — композитный скоуп, связывающий всё.
