# BECARE — каталог подводных камней формализации в Catala 1.2.0

Спутник `FORMALIZATION.md`: сюда сведены **ошибки, на которые легко наступить**, с
конкретными симптомами и исправлениями. Часть собрана на реальном прогоне case2
(«Claude Family Pro»), часть перенесена из `FORMALIZATION.md`, чтобы всё «осторожно,
тут больно» лежало в одном месте. Читай перед стартом и держи открытым во время
работы.

Общий принцип: **проверяй компилятором, а не памятью.** Синтаксис Catala заметно
менялся между версиями. Непонятную конструкцию гоняй на 10-строчном пробнике-модуле,
а не на боевом файле; после каждого блока — `make typecheck`.

---

## 1. Семантика/арифметика — самое опасное (тихо считает неверно)

Эти ловушки НЕ роняют компиляцию — они дают **молча неправильный результат**.
Только тест ловит.

- **`money * decimal` округляет к БЛИЖАЙШЕЙ копейке — до любого твоего флора.**
  Деньги в Catala хранятся как целое число копеек, и умножение денег на decimal
  сразу округляет к ближайшей копейке. Поэтому `floor(money * decimal)` —
  **no-op**: округление уже произошло. Пример боли: `2490 ₽ * 10/31` даёт `803.23`
  (ближайшее), а по договору нужен round-down → `803.22`.
  **Фикс:** флорь **decimal-произведение** и только потом переводи в деньги:
  ```
  declaration floor_to_kopeck content money depends on rub content decimal
    equals money of ((Decimal.round_by_default of (rub * 100.0)) / 100.0)
  # вызов: floor_to_kopeck of ((decimal of fee) * ratio)   -- а не floor(fee * ratio)
  ```
- **`Decimal.floor` — ложный друг: это НЕ округление вниз.** В stdlib
  `Decimal.floor of d, min_value` = `max(d, min_value)` (клэмп снизу). Округление к
  предыдущему целому — это `Decimal.round_by_default` (floor к −∞;
  `round_by_default of -3.78 = -4.0`). Для round-down неотрицательных сумм годится
  он. `round_by_excess` — вверх, `round` — к ближайшему.
- **Деление на ноль не абстрактно.** Доля `x / total` при `total = 0` (напр. разнос
  при нулевом потреблении) роняет вычисление. Явно защищай:
  `if total <= 0.0 then 0.0 else x / total`.
- **Округление вниз ⇒ остаток. Реши, кому он достаётся.** Если разносишь сумму по
  долям с round-down, сумма долей `< целого`. Не теряй копейки: отдай остаток
  явному получателю (`owner = group_charge − Σ(разнесённое)`), и проверяй инвариант
  `Σ = целому` тестом.
- **Маржинальные шкалы и «исключения из исключений» — считай на бумаге до кода.**
  Пороговые/tiered-скидки, «применяется бо́льшая из двух», «единая скидка вместо
  шкалы при превышении порога», «эта часть — по полной ставке» легко
  переплетаются. Разложи на: (а) что входит в базу, (б) какая ветка выбирается по
  какому условию, (в) от чего считается порог (от полной величины или от
  очищенной). Каждую ветку — отдельным тестом.

## 2. Синтаксис Catala 1.2.0 — роняет компиляцию

- **`output` — зарезервированное слово, НЕ имя поля.** `data output content ...`
  → `Syntax error at "output": expected the name of this struct data`.
  Переименуй (`output_regular` и т.п.).
- **Markdown-блокквоты ломают парсер.** Любая строка, начинающаяся с `>` (включая
  **перенос прозы**, случайно начавшийся с `>`, напр. `> 3 дней`), трактуется как
  директива Catala и даёт `Parsing error after token ">"`. Цитаты оформляй
  инлайн-курсивом `*"..."*`; следи за переносами строк в прозе.
- **Директивы `> Module` / `> Using` — вне ```catala-блоков.** Это строки в
  markdown-теле. Внутри code-fence → `Syntax error at ">"`. Имя модуля — с
  **заглавной** (`> Module Claude_family_pro`); имя таргета транспиляции = имя
  модуля, а не файла.
- **Структуры/функции из модели в тест-файле квалифицируй именем модуля.**
  Конструктор `Member{...}` из другого файла → `This identifier should refer to a
  struct name`. Пиши `Claude_family_pro.Member{ ... }`,
  `Claude_family_pro.TokenUsage { ... }`. То же для подскоупа:
  `fee scope Claude_family_pro.LateFeeCalculation`.
- **У подскоупа НЕТ квалификатора видимости.** `fee scope X`, а не
  `internal fee scope X` → иначе `Scope declaration does not support input
  qualifiers`.
- **Топ-левел константы НЕ экспортируются между модулями.** `OtherModule.const` →
  `unknown identifier for an external variable`. Внутри того же файла — работает.
  Решение: раздавай значение через `context`-поле с дефолтом
  (`context tariff ... / definition tariff equals const_tariff`) — тогда тестам не
  нужно тянуть константу через границу модуля (и заодно получаешь переиспользуемость).
- **`decimal of` применяется к самому близкому — ставь скобки.** Для int→decimal и
  int→money пиши `(decimal of active_days) / (decimal of days_in_month)` и
  `money * (decimal of <int>)` с явными скобками, не полагайся на приоритеты.
- **`round_by_default` уже возвращает decimal.** Не оборачивай в `decimal of (...)`
  → `to_rat on types decimal` / `type ... is unknown`.
- **Нет `let ... in` в определениях.** Промежуточные величины выноси в
  `internal`-переменные скоупа, а не пытайся связать локально.
- **`Money.min`/`Money.max` есть; для decimal min/max проще `if a<b then a else b`.**
  Проверяй наличие оператора в stdlib (`_build/libcatala/*_en.catala_en`), а не
  угадывай.
- **Неиспользуемые `internal`/`context` дают WARNING (не ошибку), но чистить стоит.**
  «Unused variable X does not contribute to any output» — убери переменную или
  задействуй; иначе шумит в каждом typecheck.

## 3. Транспиляция в Python — НЕ самодостаточна (главная неожиданность)

`make python` (`clerk build python/<Module>.py`) кладёт в `_build/python/` **один**
файл модели. Но он не запускается сам по себе:

- Он делает `from catala_runtime import *` (топ-левел) и `from . import Money_en`,
  `Date_en`, `Stdlib_en`, … (**относительные** импорты транспилированного stdlib).
- `--whole-program` это **не** спасает: даже с ним генерятся те же внешние импорты.
- Значит нужно самому собрать пакет:
  1. Транспилировать КАЖДЫЙ stdlib-модуль:
     `catala python -I _build/libcatala _build/libcatala/<x>_en.catala_en -o PKG/<Module>.py`
     (имя `<Module>` бери из `> Module` внутри файла).
  2. `_internal`-модули — это `external`, они мапятся на **рукописные** файлы
     рантайма (`decimal_internal.py` и т.п.). Скопируй их в пакет с **заглавным**
     именем (`Decimal_internal.py`, `Money_internal.py`, `List_internal.py`,
     `Date_internal.py`, `Period_internal.py`) — иначе `from . import
     Decimal_internal` не найдёт (на case-insensitive FS может «повезти», не
     полагайся).
  3. Положи `catala_runtime.py` и `dates.py` (из
     `<opam prefix>/lib/catala/runtime/python/src/catala/`) на `sys.path` (напр. в
     корень `_build/pyrun/`, а сам пакет — в `_build/pyrun/cfp/` с `__init__.py`).
  4. Транспилируй саму модель в тот же пакет.
- Автоматизируй это скриптом (`build_pyrun.sh`) и повесь на Makefile-цель
  `python-pkg`, от которой зависит `pytest`, чтобы всё воспроизводилось с чистого
  состояния. Путь рантайма ищи через
  `opam var --switch=catala prefix`, а не хардкодь.

## 4. Вызов сгенерированных скоупов из pytest — типы рантайма, не питоновские

Скоупы транспилируются в функции `snake_case` от одного `<Scope>In`-датакласса.
Аргументы — **обёртки рантайма**, а не голые Python-значения:

- `Money('2490.00')` (строка!), `Decimal(n)` (или `Decimal('1/3')`), `Integer(n)`,
  `Bool(False)`. **Голый питоновский `bool` ломается** — код зовёт `.not_()` →
  `'bool' object has no attribute 'not_'`. Всегда `Bool(...)`.
- Список (`list of T`) → **`Array([...])`** (подкласс tuple с `.map`/`.filter`), а
  НЕ питоновский `list` → иначе `'list' object has no attribute 'map'`. Хотя
  аннотация типа в коде — `List[T]`, фактически нужен `Array`.
- **`context`-вход** — это `Option[CatalaTuple[T, SourcePosition]]`. Чтобы взять
  дефолт (напр. `const_tariff`), передай `Option(None)` (Absent). Чтобы
  переопределить — `Option(CatalaTuple(value, SourcePosition(...)))`.
- **Структуры иммутабельны:** `__setattr__` кидает `NotImplementedError`. Переопределяй
  поля **при конструировании** (`Tariff(**{...})`), не присваиванием после.
- **Money = целое копеек:** для сравнений удобно `int(money)` (даёт копейки), либо
  `bool(a == b)` (операторы `==` у Value возвращают `Bool`, оборачивай в `bool()`).
- Модуль импортируется как пакет: `sys.path.insert(0, '_build/pyrun')` →
  `from cfp import Claude_family_pro as m`; `from catala_runtime import ...`.

## 5. Тулчейн и запуск тестов (раскладка «standalone», без полного clerk-проекта)

- **`clerk test` здесь НЕ линкует кросс-файловый модуль** — падает пустым «scope
  compilation failed» (корень: `Could not resolve reference to <Module>.<Scope>`).
  Рабочий путь — звать `catala` напрямую с `--whole-program`, он компилит модуль из
  исходников вместе с тест-файлом:
  ```
  catala interpret --whole-program -I . --stdlib=_build/libcatala <tests>.catala_en
  ```
- **Нужен `_build/libcatala` (рантайм+stdlib)** — создаётся `clerk start`.
- **Нативные `.cmxs` плагины stdlib** нужны для `catala interpret`: без них —
  `Compiled OCaml object "Date_internal.cmxs" not found`. Собираются как зависимость
  при `clerk build ocaml/<Module>.cmxs` (цель `setup`). После `make clean` первый
  `make test` пересоберёт их через prereq `setup`.
- **Тест = сам скоуп.** Помечай `#[test]`, фиксируй входы внутри, сверяй `assertion`.
  `catala interpret` без `-s` исполняет все input-less скоупы и проверяет их
  `assertion`. Провал → **exit 123**. Обязательно проверь негативный путь: временно
  сломай один `assertion` и убедись, что падает 123.
- **`clerk.toml` в папке модели** (хоть пустой) делает её корнем проекта — имена
  таргетов и `_build` остаются короткими. Иначе clerk берёт корнем git-репозиторий.
- `.gitignore`: `_build/`, `_targets/`, `__pycache__/`, `.pytest_cache/`.

## 6. Установка тулчейна (если с нуля)

- Catala — через **opam**: `opam switch create catala 5.2.0` → `opam install catala`.
- **Системные depexts**: `gmp ninja pkgconf` (brew). В неинтерактивном режиме opam
  **прерывается** на вопросе о depexts — ставь их заранее и добавляй
  `--assume-depexts`.
- Сборка из исходников — **10–30 минут**, запускай в фоне.
- Docker-путь (`catalalang/catala`) требует поднятого демона; opam надёжнее.

## 7. Методические грабли (из FORMALIZATION.md — не забывай)

- **Приёмочный критерий — pytest поверх Python, а не Catala-тесты.** В рантайм едет
  сгенерированный Python; если модель не транспилируется в Python — работа не
  сделана, даже если catala-интерпретатор её прогоняет. `ocaml`/`c` — второстепенны.
- **Состояние между периодами — не дело Catala.** Остаток/перенос, счётчик стажа,
  флаг реферала, доступность SLA, ставка НДС (битемпоральное версионирование) —
  это **входы от рантайма** (`input`), в pytest — **моки**. Для каждого такого
  входа опиши в прозе `.catala_en`: что это, откуда приходит, в каких единицах.
- **Формализация фиксирует интерпретацию.** Юртекст двусмыслен (границы «< / ≤»,
  «после N-го», от чего считается порог). Выбери трактовку, **явно** запиши в прозе
  рядом с кодом и покрой **граничным** тестом (напр. значение ровно на пороге).
- **Пиннь версии, а не значения.** Все ставки/пороги — в одной структуре и одной
  константе с провенансом (источник, стороны, конкретные числа); скоупы берут её
  через `context` с дефолтом → переиспользуемость + объяснимость + ретро-пересчёты.
- **Защищай инварианты явно** (`if x < 0 then 0`, потолки, границы), не полагайся на
  «естественную» арифметику.
- **Прозаические пункты — вне модели, но помечены.** Обязанности, процедуры,
  реквизиты — оставляй прозой как контекст, не тащи в скоупы; явно отметь, что вне
  модели.

## 8. Definition of Done (чек-лист)

- [ ] `make typecheck` — `Typechecking successful!`, exit 0, без WARNING.
- [ ] Каждая ветка `if/then/else` покрыта тестом; негативный путь даёт exit 123.
- [ ] Каждая зафиксированная интерпретация — в прозе И в граничном тесте.
- [ ] Провенанс (документ, стороны, значения) — в модели, в одной константе.
- [ ] Каждый вход от рантайма описан в прозе (что/откуда/единицы).
- [ ] Прозаические пункты явно отмечены как вне модели.
- [ ] **`make python`** собирает Python-артефакт без ошибок.
- [ ] **`make pytest`** зелёный: тесты поверх собранного Python-пакета, входы
      рантайма замоканы, границы каждой ветки покрыты, инварианты проверены.
- [ ] Всё воспроизводится с `make clean`.
